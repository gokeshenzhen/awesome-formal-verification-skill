# FPV Benchmark Results — 2026-06-23

First task-level eval run, motivated by the question: *does the lossy
many-docs → one-file distillation hurt skill quality, and where?*

Method: `scripts/loss_probe.py` (content-coverage screen) found candidate losses;
`benchmarks/run_scenarios.sh` then ran each scenario through a headless agent
(`claude -p`, formal-verification skill installed) and graded CONCEPT tokens
(general methodology) separately from EXACT tokens (tool-specific JG commands).

## Results

| Scenario | Type | Concept | Exact (JG) | Verdict |
|---|---|---|---|---|
| counter-abstraction-timeout | control | ✅ counter-abstraction, milestone | ✅ `abstract -counter` (4×) | strong pass |
| proof-decomposition-ag | control | ✅ assume-guarantee, helper, sound | — (none) | strong pass |
| config-logic-cutpoint | loss-probe | ✅ cutpoint, legality, assumption | ❌ `setup_ndc` (0×) | concept recovered, exact lost |
| per-property-simplification | loss-probe | ✅ per-property, simplification | ❌ `set_per_property_simplification` (0×) | concept recovered, exact lost |

## Finding

**The distillation loss is real but narrow, and the coverage diff over-counts it.**

- **Controls pass strongly.** Where the knowledge file keeps the content, the agent
  answers with tool-accurate detail (`abstract -counter`, milestones, sound AG).
- **General methodology is NOT lost in practice.** Both loss-probe scenarios dropped
  their technique from the knowledge file (`loss_probe` flagged `cutpoint`,
  `set_per_property_simplification`), yet the agent still produced the right concept
  (cutpoints + legality assumptions; per-property simplification) — the model supplies
  general FV knowledge itself.
- **Tool-specific atoms ARE lost.** The exact JG commands the model can't derive —
  `setup_ndc`, `set_per_property_simplification` — were missing from the answers,
  because they were dropped from the knowledge file and the model doesn't know them.

**Implication for the design:** distillation should be aggressive on general prose
(the model recovers it) and conservative on **tool-specific atoms** — exact command
names, flags, and numeric thresholds. Progressive-disclosure sub-files for the rich
modules should prioritize preserving those atoms, not re-explaining methodology.

## Reproduce

```bash
bash scripts/install.sh            # skill must be installed
bash benchmarks/run_scenarios.sh   # all scenarios (re-runs claude -p)
python3 scripts/loss_probe.py      # the content-coverage screen (needs local extractions/)
```
